# AI 修仙灵宠

一台拥有身体、生活、记忆与长期成长的便携式 AI 修仙电子宠物。

设备以 ESP32-S3、480×480 AMOLED 触摸屏、实体按键、麦克风、扬声器、Wi-Fi、Bluetooth LE、IMU、RTC 和电池为硬件基础。游戏以“随身洞府、多世道统、异步仙缘”为核心，通过自有服务端接入 LLM，使灵宠能够对话、记忆共同经历、形成持续人格并主动参与玩家的现实生活与修炼旅程。

当前项目处于 **V0 原型阶段**。第一目标不是一次性完成完整游戏，而是验证动画表现、语音互动、网络链路、续航和每日使用价值。

## 核心方向

- **随身洞府**：灵宠会在洞府中修炼、炼丹、练剑、休息和探索，不操作时也有自己的生活。
- **多世道统**：每次转生留下本命神通、生平、性格与前世痕迹，后世可以遇见前世残魂。
- **LLM 神魂**：模型负责理解、表达、人格和动态叙事；本地游戏引擎负责数值、奖励、存档和安全边界。
- **语音陪伴**：按住说话，设备采集语音，经服务端完成识别、推理和合成后流式返回。
- **现实联动**：玩家的工作、专注、日程、时间和天气可以转化为修仙事件。
- **异步仙缘**：通过 Wi-Fi 和 Bluetooth LE 实现飞剑传书、洞府拜访、留影切磋和近距离论道。
- **离线可用**：断网时保留本地游戏、基础人格、预置台词、动画、存档和事件模板。

## 硬件基线

首版原型及后续开发固定使用已购入的完整硬件平台 **Waveshare ESP32-S3-Touch-AMOLED-2.16**，不再评估替代主板：

- ESP32-S3R8，双核最高 240MHz
- 8MB PSRAM、16MB NOR Flash
- 2.16 英寸 480×480 AMOLED 电容触摸屏
- 2.4GHz Wi-Fi、Bluetooth LE 5
- 双麦克风、ES7210 采集与回声消除
- ES8311 音频播放及扬声器输出
- QMI8658 六轴 IMU
- PCF85063 RTC
- TF 卡槽
- AXP2101 电源管理
- 3.7V 1000mAh 锂电池
- 四个外接实体游戏按键

设备不使用震动马达。受击、暴击、突破和通知通过 AMOLED 动画、音效、语音和实体按键反馈表达。

完整采购、连接和验收说明见 [硬件清单](docs/hardware-bom.md)。

## 固件开发框架

本项目固定使用 **ESP-IDF** 开发 ESP32-S3 固件。Arduino 示例仅用于参考微雪官方外设初始化方式，不作为项目框架，也不引入 Arduino 兼容层。界面使用 LVGL，并优先复用微雪官方 ESP-IDF BSP、驱动和示例完成首轮硬件验证。

唯一 Git 仓库是 `immortal_pet_v2`；实际 ESP-IDF 工程是 `firmware/device`。在 VS Code 中打开 `G:\code\immortal_pet_v2\firmware\device` 后，使用 ESP-IDF 的 Build、Flash 和 Monitor。该目录保留小智官方的工程结构，避免额外的 CMake 包装层。

`firmware/device/sdkconfig` 已固定为 ESP32-S3、Waveshare 2.16 英寸 AMOLED、设备端 AEC 和 `IMMORTAL_PET_V2`。

智能语音固定接入 [78/xiaozhi-esp32](https://github.com/78/xiaozhi-esp32)。小智负责唤醒、录音、AEC、Opus 音频流、语音会话、服务端通信和 MCP 能力；本项目保留修仙灵宠 UI、游戏状态、数值、存档、人格记忆和安全边界的控制权。小智主线已经提供 `waveshare/esp32-s3-touch-amoled-2.16` 专用板卡适配，首轮以该适配跑通原厂语音链路，再逐步接入游戏层，不从零重写音频底层。

详细边界和实施顺序见 [小智 AI 语音接入方案](docs/xiaozhi-ai-integration.md)。

当前小智服务端需要配置为灵宠第一人称，配置文本见 [灵宠智能体角色配置](docs/xiaozhi-pet-persona.md)。小智仅作为底层语音链路，不作为产品入口或可见角色。

玩法循环、首页入口和功能解锁顺序见 [游戏玩法与入口规划](docs/gameplay-and-entrypoints.md)。

## 系统架构

```text
┌──────────────────────────────────────────────────────────┐
│  ESP32-S3 设备                                            │
│                                                          │
│  本地游戏引擎 ── UI/动画 ── 音频 ── 输入 ── 存档          │
│       │             │          │       │                  │
│       └──────────── 网络客户端 / 离线降级 ───────────────┐ │
└────────────────────────────────────────────────────────┬─┘
                                                         │ TLS
                                                         ▼
┌──────────────────────────────────────────────────────────┐
│  自有服务端                                               │
│                                                          │
│  设备鉴权 ── LLM 网关 ── 长期记忆 ── 语音服务 ── OTA      │
└───────────────────────────┬──────────────────────────────┘
                            │
                            ▼
                  模型服务 / 天气 / 日历 / AI Hook
```

关键边界：

- 模型服务商 API 密钥只保存在服务端，不写入固件和设备文件。
- LLM 只能返回经过定义的对白、情绪、动画和事件建议。
- 经验、资源、战斗、解锁、存档等结果由确定性游戏引擎计算。
- 服务端和设备端均校验 LLM 输出，拒绝未注册动作和越权修改。
- 没有网络时，设备仍然是一台完整的本地电子宠物。

## V0.1 原型范围

首轮只验证以下闭环：

1. AMOLED 显示、触摸和基础 LVGL 页面。
2. 一只灵宠的待机、行走、睡眠、修炼、受击和突破动画。
3. 四个实体按键及按住说话。
4. 麦克风录音、服务端上传和扬声器播放。
5. Wi-Fi 配网、BLE 近距离发现和 TLS 通信。
6. 电池电量、充电、休眠、唤醒和真实续航。
7. 一个稳定人格的文字与语音对话。
8. 问候、工作结算、夜间回顾三类结构化 LLM 事件。
9. 网络断开时的本地台词、动画和存档降级。
10. 桌面模拟器和一条自动化验证路径。

V0.1 不做多人实时战斗、复杂装备、开放世界、摄像头识别、本地大模型和定制 PCB。

## 开发原则

- 固件统一使用 ESP-IDF，UI 使用 LVGL；不使用 Arduino 作为项目框架。
- 本地游戏逻辑必须确定、可测试，不依赖 LLM 正确发挥。
- 协议采用版本化 Schema，设备拒绝未知字段和未授权动作。
- 网络、音频、动画和存储使用独立任务，避免阻塞主循环。
- 大型动画、语音和音乐资源放入 TF 卡，Flash 只保存启动必需资源。
- 每项在线功能都要有明确的离线行为。
- 先测真实帧率、功耗、温升和延迟，再决定是否定制 PCB。
- 不依赖卖家预装的封闭“小智AI/DeepSeek”服务；只复用开源 xiaozhi-esp32 客户端能力，正式服务端、账户、记忆和游戏规则由本项目控制。

## 与第一代项目的关系

第一代 RP2040 项目位于 `claude_code_pet`，作为已完成的稳定设备保留。本项目是独立的第二代产品，不兼容第一代固件、二进制存档、串口协议和 128×160 UI。

可以继承的经验包括：

- 非阻塞游戏状态机
- A/B 存档和校验保护
- 固件规则与参考模型同步测试
- 素材转换管线
- 桌面模拟器
- AI 工作事件与游戏成长结合

新项目不会直接复制第一代工程结构，而是根据 ESP32-S3、联网、音频和 LLM 场景重新设计。
